Asterinas · AArch64 port

QEMU machine-type compatibility

Every one of qemu-system-aarch64's 109 machine types, booted with the same full Asterinas kernel and initramfs. A pass means the kernel reached userspace and a signal handler returned through the vDSO. Boards that demand multiple cores were re-run with the cores they require — SMP is verified working (4 cores boot, schedule, and do TLB shootdowns). The port targets the paravirtual virt board.

SMP is not the blocker. 33 machines were first flagged “requires SMP” only because the harness passed -smp 1. Re-run with the cores each board demands, none failed on SMP: they are 32-bit ARMv7 SoCs (image rejected), Cortex-M microcontrollers, or genuine AArch64 boards that load but lack a PL011 UART at virt's 0x0900_0000. The 12 blue rows — incl. raspi3b, raspi4b, and sbsa-ref — are the real next-target candidates.

Base run: -cpu cortex-a72 -m 2G -kernel <elf> with the initramfs via -device loader. Boards that reject that CPU or demand a core count were re-run with their native CPU and -smp N (N shown per row). A machine outside virt fails because it is different hardware — a 32-bit CPU, a Cortex-M core, a fixed RAM size, or a distinct SoC memory map with no PL011 at 0x0900_0000 — not because of any missing kernel capability. This mirrors how the upstream x86 / RISC-V / LoongArch ports each target one board.

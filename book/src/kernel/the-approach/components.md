# Components

A component is a Rust crate that participates in Asterinas's staged
initialization framework. It depends on the `component` crate, is declared in
the workspace `Components.toml`, and can submit initialization functions with
the `#[init_component]` attribute.

Component crates remain ordinary Cargo crates. Their `Cargo.toml` dependencies
define what code is compiled, linked, and callable. The component framework
adds initialization discovery and ordering; it does not replace Cargo's
dependency model.

## Initialization hooks

A component can declare at most one initialization function for each stage:

- `#[init_component]` or `#[init_component(bootstrap)]` registers a
  `Bootstrap` hook. It runs on the bootstrap processor after OSTD
  initialization and before the core subsystem initialization.
- `#[init_component(kthread)]` registers a `Kthread` hook. It runs from the
  first non-idle kernel thread.
- `#[init_component(process)]` registers a `Process` hook. It runs when the
  first user process starts.

Each hook has the following shape:

~~~rust
use component::{ComponentInitError, init_component};

#[init_component(kthread)]
fn init() -> Result<(), ComponentInitError> {
    // Initialize state that requires kernel-thread context.
    Ok(())
}
~~~

The component macros collect these hooks at link time. For each stage,
`aster-core` calls `component::init_all` with metadata generated by
`component::parse_metadata!`.

Within a stage, the framework derives priorities from the Cargo dependency
graph. If component A depends on component B, B's hook runs before A's hook in
that stage. A hook error is reported through the boot log and initialization
continues with the remaining hooks. Errors that prevent the framework from
matching component metadata are returned by `init_all`; the current boot path
treats those errors as fatal with `unwrap()`.

## Current and planned component placement

Components are classified by their position relative to `aster-core`.

| Kind | Location | Dependency rule | Current state |
|------|----------|-----------------|---------------|
| Low-level | `kernel/core/comps/` | May be named by `aster-core`; must not depend on `aster-core` | Contains all existing component crates |
| High-level | `kernel/comps/` | May depend on `aster-core`; must not be named by the core or lower crates | Planned for later phases; no high-level component or generic selection mechanism exists yet |

A crate without component initialization is normally a library under
`kernel/libs/`. Code belongs directly in `aster-core` only when code at or
below the core must call it by name and the dependency cannot be inverted
through an interface.

This classification is a dependency rule, not only a directory convention.
Cargo must be able to form an acyclic graph. A low-level component cannot start
using core services while the core still depends on it; moving it above the
core requires a separate migration of its dependencies and interfaces.

## Planned interfaces across layers

The rules in this section constrain future high-level component migrations.
They do not describe a generic cross-layer interface or registry facility
implemented by this crate reorganization.

A future high-level component would be able to call exported `aster-core`
APIs through its normal downward Cargo dependency. The reverse direction
cannot be a direct call to the high-level crate because that would introduce a
cycle.

When a future migration requires lower code to invoke behavior implemented by
a higher component, the lower layer must own the interface. A trait and typed
registry are one possible form already used by some device subsystems. The
higher component would implement and register that interface during
initialization, allowing the lower layer to call it without naming the
implementing crate.

Asterinas already applies this pattern to several device registries. The crate
reorganization does not introduce a universal registry, and the exact
interface for each future high-level component must be designed and reviewed
with that component.

## Current inclusion and metadata

Today, all existing low-level components are direct or transitive dependencies
of `aster-core`, so they are included with the core. There are no high-level
components and no generic assembler-level component selection mechanism in the
current tree.

For future high-level components, Cargo dependencies or features can make a
crate available to the assembler, but that alone may not ensure that a crate
containing only link-time registrations is retained in the final kernel. Such
components will also require explicit assembler linkage or wiring. The exact
selection mechanism must be implemented and validated when the first
high-level component is introduced.

The root `Components.toml` records component package names used by
`parse_metadata!`. The macro checks workspace component crates against this
file and uses Cargo metadata to compute initialization priorities. The file
does not add a dependency, choose a component implementation, or load code at
runtime.

## Current limits

Components are build-time units. Runtime loading, unloading, and replacement
are not provided by the current component framework.

The separate `cargo-component` tooling contains experimental
component-level access-control support. This crate reorganization neither
enables nor redesigns that mechanism. Access-control policy, component
initialization, and Cargo-based component selection are distinct concerns.

All kernel components remain on the safe-Rust side of the framekernel
boundary. Operations that require unsafe Rust belong in OSTD behind sound safe
APIs.
